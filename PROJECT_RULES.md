# 项目协作规则

本项目按 SDD（Spec-Driven Development，规格驱动开发）方式推进。任何功能、模型、数据、界面或部署相关改动，都应遵循以下规则。

## 1. 先设计，再讨论，再改动

- 变动之前先讨论需求、边界和影响范围。
- 不在需求尚未澄清时直接改核心逻辑。
- 对模型、数据源、用户流程、部署方式等影响较大的改动，必须先说明方案和风险。

## 2. 讨论时给出建议

- 讨论问题时，不只罗列选项，也要给出推荐方案。
- 建议应说明取舍，例如用户体验、实现成本、数据可信度、维护成本和安全风险。
- 如果存在短期 MVP 做法和长期产品化做法，应明确区分。

## 3. 讨论结束后先形成文档

- 重要结论先写入项目文档，再进入实现。
- 新需求优先记录到 `specs/TODO.md`。
- 已确定要实现的能力，应同步更新相关规格文档，例如：
  - `specs/PRODUCT_SPEC.md`
  - `specs/UX_SPEC.md`
  - `specs/AI_SPEC.md`
  - `specs/DATA_SPEC.md`
  - `specs/IMPLEMENTATION_PLAN.md`

## 4. 再进行代码改动

- 代码改动应能追溯到已讨论和已记录的规格。
- 实现时保持改动范围清晰，不把无关重构混入同一批任务。
- 完成后运行必要检查，并说明验证结果。

## 5. 同步更新 README

- 用户可感知的能力变化，需要同步更新 `README.md`。
- README 应反映当前真实状态，而不是早期 MVP 状态。
- 以下变化尤其需要同步：
  - 项目定位变化。
  - 数据源变化。
  - 模型参数或默认值变化。
  - LLM 参与边界变化。
  - 启动方式、部署方式或交付形式变化。
  - 面向普通用户的操作流程变化。

## 6. 提交前检查

- 提交前确认工作区中没有无关文件。
- 代码改动应至少运行对应的语法或基础功能检查。
- 文档改动应检查术语一致性，例如“夺冠预测”“AI 信息因子”“赛程信息”等命名。

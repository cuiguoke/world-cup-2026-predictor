### 5. 本地预测模型复盘与改进

目标：

- 系统性讨论当前本地预测模型是否合理，避免模型结果只是参数和数据偏差的产物。
- 明确哪些参数应暴露给用户，哪些应保持为专家设置。

待讨论问题：

- 32 强淘汰赛当前使用公开签位结构，并用确定性贪心方式分配成绩最好的小组第三名；后续是否应替换为 FIFA 官方小组第三名分配表。
- Elo 初始分、K 值、赛事权重、时间衰减是否合理。
- 东道主加成是否过高，是否应按比赛地点或阶段区别处理。
- Poisson 进球模型是否需要加入进攻/防守拆分，而不是只使用单一 Elo 差。
- 小组赛和淘汰赛是否应使用不同的保守程度。
- 点球大战是否应加入独立参数，而不是只使用 Elo 期望胜率。
- 是否纳入最终名单、伤病、休息天数、旅途消耗和战术克制等比赛上下文。
- 是否纳入博彩公司赔率或球员级别实力数据；如果纳入，如何处理数据可信度、时效性和版权/来源限制。
- 完整历史数据中球队名称、历史国家队变化、友谊赛质量是否会造成偏差。
- 是否需要提供“无主场加成”“低主场加成”“高主场加成”等对照预测。

建议：

- 增加模型参数说明页，解释当前默认值。
- 增加若干参数预设，而不是让普通用户直接面对所有公式参数。
- 输出模型诊断信息，例如前 20 名 Elo、东道主加成影响、数据来源范围。
- 用历史世界杯做回测，比较不同参数组合的合理性。
- 将 README 中“局限与后续改进”的内容同步维护到 TODO，避免后续 README 重写时丢失。

验收：

- 能解释主要模型参数对预测结果的影响。
- 能生成至少一组对照预测，展示东道主加成或参数变化带来的差异。
- README 和报告中说明模型局限。

### 7. 按比赛阶段组织预测卡片

目标：

- 除小组赛出线预测外，根据世界杯后续比赛安排，按赛事阶段展示不同预测内容。
- 让普通用户不需要理解 `r32`、`r16`、`qf`、`sf` 等缩写，也能按阶段阅读预测。

建议：

- 在“夺冠预测”中增加阶段卡片：
  - 小组赛：小组头名、前二、晋级 32 强概率。
  - 32 强：预计对阵、晋级 16 强概率。
  - 16 强：晋级八强概率。
  - 八强：晋级四强概率。
  - 四强：晋级决赛概率。
  - 决赛：夺冠概率。
- 阶段卡片应随赛事进展更新：已结束阶段显示结果，未结束阶段显示预测。
- 导出的 HTML 报告也按阶段组织，而不是只给总表。

验收：

- 用户能按比赛阶段浏览预测，不需要理解工程字段名。
- 已录入比分会改变当前阶段和后续阶段预测。
- 每张阶段卡片都显示预测时间、模拟次数和主要不确定性。

### 8. 支持 AI 辅助录入比分

目标：

- 减少用户逐场手动输入比分的负担。

建议：

- 用户粘贴赛果文本，或提供权威赛果 URL。
- 后端抓取正文后调用 LLM。
- LLM 提取待确认比分列表。
- 系统用正式赛程匹配球队和比赛。
- 用户确认后写入比分并重新预测。

验收：

- 能从一段文本中识别多场比分。
- 能识别中文队名、英文队名和常见别名。
- 低置信度或无法匹配的结果不会自动写入。
- 已有比分冲突时提示是否覆盖。

### 12. 多用户后台逻辑优化

目标：

建议：

验收：
